The npm wrapper that makes the Ductus Dart adapter runnable for [`@ductus/core`](https://github.com/PlaxXOnline/ductus/tree/main/packages/core) ‚Äî the actual code analysis lives in the pub.dev package [`ductus`](https://pub.dev/packages/ductus) and runs in the Dart toolchain.

This package contains **no analysis logic**. It only provides the `ductus-adapter-dart` binary, which invokes `dart run ductus:adapter` with the appropriate package context, passes stdout/stderr and the exit code through, and guarantees: **stdout is exactly one graph JSON document** (pub preamble such as `Resolving dependencies...` is redirected to stderr, nothing is lost).

**Prerequisite:** an installed [Dart SDK](https://dart.dev/get-dart) on the `PATH` (already present in Flutter projects). Node.js ‚â• 20.

## Installation

```bash
npm install --save-dev @ductus/core @ductus/adapter-dart
```

In addition, `ductus:adapter` must be resolvable on the Dart side ‚Äî either of these two options is enough:

```bash
# Option 1: in the target project (recommended, versioned with the project)
dart pub add dev:ductus

# Option 2: globally, with no entry in the target project at all
dart pub global activate ductus
```

If you import the Dart annotations (path B, see below) in `lib/`, add `ductus` as a regular dependency instead: `dart pub add ductus`.

## Quickstart with @ductus/core

```bash
npx ductus init       # creates ductus.config.yaml
npx ductus extract    # invokes the Dart adapter ‚Üí journey-graph.json
npx ductus generate   # LLM (BYOK) ‚Üí end-user docs as MDX or a website
```

The relevant excerpt of `ductus.config.yaml` (as generated by `ductus init`):

```yaml
adapters:
  - dart:
      project: .
      deriveFrom: [go_router, auto_route]
```

For more on configuration, LLM providers and output formats, see the README of [`@ductus/core`](https://github.com/PlaxXOnline/ductus/tree/main/packages/core).

## How @ductus/core finds the adapter

`ductus extract` resolves the command for the `dart` adapter entry in this order (implemented identically in the core runner and in this wrapper):

| # | Source | Behavior |
|---|--------|----------|
| 1 | `command:` in the adapter entry of `ductus.config.yaml` | Always wins ‚Äî the configured command is executed as-is. |
| 2 | `ductus-adapter-dart` binary (this package) | Looked up in `node_modules/.bin` next to the config, then on the `PATH`. Internally, the wrapper continues the chain with steps 3‚Äì5. |
| 3 | Environment variable `DUCTUS_DART_ADAPTER_DIR` | `dart run ductus:adapter` with this directory as the working directory ‚Äî useful when neither the project nor pub-global knows the package (e.g. a monorepo checkout). |
| 4 | The target project's `pubspec.yaml` declares `ductus` | `dart run ductus:adapter` directly in the target project (`dependencies` or `dev_dependencies`). |
| 5 | Globally activated package (`dart pub global activate ductus`) | `dart pub global run ductus:adapter`; for a path activation, `dart run` runs directly in the source directory. |

If no step applies, the invocation aborts with an error message that lists the options (`dart pub add dev:ductus`, `dart pub global activate ductus` or `DUCTUS_DART_ADAPTER_DIR`). So the target project needs **no** `ductus` dependency as long as one of the other steps applies.

## Which sources the adapter understands

The Dart adapter combines four input paths into one graph; manual annotations override derived values field by field:

| Path | Source | Dependency in the target project? |
|------|--------|-----------------------------------|
| A | Comment convention `@journey:screen`, `@journey:action`, `@journey:decision`, `@journey:flow` in `//`/`///` comments | None ‚Äî completely build-free |
| B | Dart annotations `@JourneyScreen`, `@JourneyAction`, `@JourneyDecision`, `@JourneyFlow` | `ductus` (import only, no runtime behavior) |
| C | Automatic derivation from `go_router` (`GoRoute` ‚Üí screens, `ShellRoute` ‚Üí flows, `redirect:` ‚Üí decisions, `context.go()/push()/‚Ä¶` ‚Üí transitions) and from `auto_route` (`@RoutePage()` ‚Üí screens, best effort) | None (only the router package itself) |
| D | build_runner artifact `ductus_builder.g.json` ‚Äî the builder from the `ductus` package runs as a build step and also resolves non-literal constants; the adapter passes the artifact through with `--from-builder` or the config key `fromBuilder: true` | `ductus` + `build_runner` |

Details, examples and best practices for all four paths: [README of the `ductus` package](https://pub.dev/packages/ductus). Two runnable demo apps (path A only, and paths B+C combined) live under [examples/](https://github.com/PlaxXOnline/ductus/tree/main/examples).

## Direct invocation (optional)

Normally `ductus extract` starts the wrapper automatically. Manually:

```bash
ductus-adapter-dart --project <dir> [--config <json-file>] [--no-debug-file] [--from-builder]
```

- `--project <dir>` (required): the Dart/Flutter project to analyze.
- `--config <json-file>`: adapter configuration as a JSON object with the keys `deriveFrom` (default `["go_router", "auto_route"]`), `include` (glob patterns relative to the project, default `["lib/**"]`) and `fromBuilder` (default `false`). `@ductus/core` generates this file automatically from the adapter entry in `ductus.config.yaml`.
- `--no-debug-file`: suppresses the debug file `ductus_graph.g.json` in the project directory.
- `--from-builder`: passes the build_runner artifact `ductus_builder.g.json` through instead of scanning itself (path D; equivalent to the config key `fromBuilder: true`, the flag wins).

stdout is exactly one canonical graph JSON; warnings and diagnostics go to stderr. The Dart adapter's exit code is passed through unchanged.
