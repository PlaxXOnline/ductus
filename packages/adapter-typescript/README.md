# @ductus/adapter-typescript

**English** | [Deutsch](./README.de.md) | [Espa√±ol](./README.es.md) | [ÁÆÄ‰Ωì‰∏≠Êñá](./README.zh-CN.md)

**End-user documentation straight from your TypeScript/JavaScript code.**
The Ductus adapter for TS/JS projects extracts a user-journey graph from
`@journey:` comments and react-router/Next.js configurations ‚Äî the
[Ductus CLI (`@ductus/core`)](https://github.com/PlaxXOnline/ductus/tree/main/packages/core)
turns it into polished docs via LLM (BYOK), as MDX files or a static
website, versioned alongside your code.

- **Build-free, no dependency in the target project:** the adapter parses
  the sources through the TypeScript compiler API (parse-only) ‚Äî no
  `npm install` in the target project, no build, no tsconfig required.
- **Two input paths, freely combinable:** `@journey:` comments (path A,
  syntax identical to the Dart adapter) and automatic derivation from
  react-router or Next.js (path C).
- **TS and JS, with and without JSX:** scans `.ts`, `.tsx`, `.mts`,
  `.cts`, `.js`, `.jsx`, `.mjs`, `.cjs`.
- **Deterministic:** the output is canonical, byte-stable JSON ‚Äî ideal
  for code review and CI.

## Installation

Prerequisite: Node.js ‚â• 20 ‚Äî nothing else, the adapter runs entirely in Node.

```bash
# in the project (recommended, versioned with the project)
npm install --save-dev @ductus/core @ductus/adapter-typescript

# or globally, with no entry in the target project at all
npm install -g @ductus/core @ductus/adapter-typescript
```

## Quickstart with @ductus/core

```bash
npx ductus init       # detects package.json, creates ductus.config.yaml
npx ductus extract    # invokes the adapter ‚Üí journey-graph.json
npx ductus generate   # LLM (BYOK) ‚Üí end-user docs as MDX or a website
```

`npx ductus help <command>` shows the options of each command. The relevant
section of `ductus.config.yaml` (as generated by `ductus init`):

```yaml
adapters:
  - typescript:
      project: .
      deriveFrom: [react-router, next]
```

`ductus init` reads the `package.json`: `app.name` becomes the package name,
and `deriveFrom` lists the routers found under `dependencies` or
`devDependencies` (`react-router`/`react-router-dom` ‚áí `react-router`,
`next` ‚áí `next`); if none is found, the default `[react-router, next]`
stays in place. If a `pubspec.yaml` exists in the same directory, it takes
precedence ‚Äî Flutter projects often carry a `package.json` for tooling.

More on configuration, LLM providers and output formats in the README of
[`@ductus/core`](https://github.com/PlaxXOnline/ductus/tree/main/packages/core).

## Build-free: the `@journey:` comment convention

The manual input path (path A) ‚Äî syntax and semantics identical to the
[Dart adapter](https://github.com/PlaxXOnline/ductus/tree/main/dart/ductus),
works in `//` line comments:

```tsx
// @journey:flow id="auth" title="Sign-in" start="login"

// @journey:screen id="login" title="Sign-in" flow="auth"
//   description="Screen where the user signs in."
export function LoginPage() {
  // @journey:action label="Sign in" to="dashboard" trigger="submit"
  //   condition="credentials valid"
  const onSubmit = () => { /* ‚Ä¶ */ };
  return /* ‚Ä¶ */;
}
```

A block starts with `@journey:<screen|action|decision|flow>`, pairs are
`key="value"` (`\"` escapes a quote), continuation in immediately following
comment lines; it ends at the first non-comment line or at the next
`@journey:` block. Unknown keys and triggers are ignored with a warning
(an unknown `trigger` falls back to `tap`); missing required fields abort
the run with an error. `tags` is comma-separated.

| Block | Keys (required **bold**) | Effect on the graph |
|---|---|---|
| `@journey:screen` | **`id`**, **`title`**, `flow`, `description`, `tags` | Screen node |
| `@journey:action` | **`label`**, **`to`**, `from`, `id`, `trigger`, `condition` | Transition (edge) |
| `@journey:decision` | **`id`**, **`title`**, `flow`, `description`, `tags` | Decision node (branch point) |
| `@journey:flow` | **`id`**, **`title`**, **`start`**, `description` | Named flow; `start` must be the id of a screen |

- `trigger` is one of `tap` (default), `submit`, `auto`, `back`,
  `deeplink`, `system`.
- **Component binding:** `screen`/`decision` blocks bind to the enclosing
  or immediately following component ‚Äî a top-level class, function
  declaration or `const` with a function initializer, including
  `memo(‚Ä¶)`/`forwardRef(‚Ä¶)` wrappers.
- If a `@journey:action` has no `from`, the enclosing component that is a
  known screen (via `@journey:screen` or from derivation) applies; if no
  screen can be determined that way, the run aborts with an error.
- Without an action `id`, `e_<from>_<to>` is generated deterministically.

Typed annotations and a builder (paths B and D of the Dart adapter) are
deliberately absent here ‚Äî TypeScript doesn't need them, path A is the
manual path and stays free of any dependency in the target project.

## Automatic derivation from react-router / Next.js

Even without a single annotation you already get a useful graph (path C).
Which derivations run is controlled by `deriveFrom` (default: both).

### `react-router`

Recognized are object routes from `createBrowserRouter`, `createHashRouter`,
`createMemoryRouter` and `useRoutes` ‚Äî as an inline array or as a route
constant in the same file (`const routes = [‚Ä¶];
createBrowserRouter(routes)`) ‚Äî as well as `<Route>` JSX (which also covers
`createRoutesFromElements`):

| Source | becomes |
|---|---|
| Route with a path | Screen node |
| Pathless layout route with children | Flow `shell-N` (`start` = first child screen) |
| `loader` (inline or declared in the same file) that calls `redirect('‚Ä¶')` | Decision node `<screen>_redirect` with `auto` edges |
| `<Link to>` / `<NavLink to>` | Transition (`tap`; `label` = sole text child) |
| `<Navigate to>` | Transition (`auto`) |
| `navigate('‚Ä¶')` ‚Äî only in files that use `useNavigate` | Transition (`tap`) |

Nested paths are joined absolutely (`path: 'detail'` under `/users`
‚áí `/users/detail`). `element={<X />}` or `Component={X}` associates the
component with the screen ‚Äî a single wrapper such as `<Suspense>` is seen
through; this association is how the adapter finds the `from` of the
navigation edges.

### `next`

File-based routing, App and Pages Router (each also under `src/`):

| Source | becomes |
|---|---|
| App Router: `app/**/page.*` | Screen node |
| Route group `(name)/` | Flow `name` (`start` = first page of the group) |
| Pages Router: `pages/**` (excluding `_app`/`_document`/`_error` and `api/`) | Screen node |
| `redirect('‚Ä¶')` / `permanentRedirect('‚Ä¶')` in a page file with a `next/navigation` import | Decision node `<screen>_redirect` with `auto` edges |
| `<Link href>` | Transition (`tap`; `label` = sole text child) |
| `router.push('‚Ä¶')` / `router.replace('‚Ä¶')` ‚Äî only in files that use `useRouter` | Transition (`tap`) |

`@slot` parallel routes, `(.)` intercepting routes and `_private` folders
are not standalone targets and are skipped.

Pages-Router screens are only created with **Next evidence**: `next` in the
`dependencies`/`devDependencies` of the `package.json`, a `next.config.*`,
or an import from `next`/`next/‚Ä¶` in the scanned sources. Without evidence,
`(src/)pages/` stays silent ‚Äî the folder name is a common convention in
react-router projects too and would otherwise produce phantom screens. The
App Router (`app/**/page.*`) is unambiguous as a convention and needs no
evidence.

### Derived ids and notes

- The screen id is the route's `id:` property (react-router) or the path
  slug: leading `/` is dropped, `/` ‚Üí `-`, parameter segments (`:id`,
  `[id]`) are dropped, empty path ‚áí `root` ‚Äî `/users/:id/edit` ‚áí
  `users-edit`. The `title` is the humanized id (`users-edit` ‚áí
  ‚ÄúUsers edit‚Äù).
- Derived nodes and edges carry `source: "derived"` and a `sourceRef`
  pointing to where they were found (flows have no `source`/`sourceRef`
  in the schema).
- Unresolvable targets or sources (the path matches no known route, the
  enclosing component is no known screen) are not an error: the adapter
  writes a note to stderr and drops the edge.

## Merge rules

Identical to the Dart adapter: manual `@journey:` entries override derived
values **field by field** ‚Äî provided they use **the same id** (the derived
ids are in `journey-graph.json` after `ductus extract`). If two **manual**
sources contradict each other, the run aborts fail-fast citing both
locations. Edges without an explicit id get `e_<from>_<to>`
deterministically (on collisions `_2`, `_3`, ‚Ä¶); the output is canonically
sorted and byte-stable across repeated runs, `meta.adapters` contains
`[{ "name": "typescript", "version": ‚Ä¶ }]`.

## The adapter CLI

Normally `ductus extract` starts the adapter automatically. Manually:

```bash
ductus-adapter-typescript --project <dir> [--config <json-file>] [--no-debug-file]
```

| Option | Meaning |
|---|---|
| `--project <dir>` | Project directory (required) |
| `--config <json-file>` | JSON configuration file (keys below); `@ductus/core` generates it automatically from the adapter entry in `ductus.config.yaml` |
| `--no-debug-file` | Suppresses the debug file `ductus_graph.g.json` in the project directory (default: it is written) |

Behavior: stdout is exactly one canonical graph JSON, all diagnostics
(warnings, notes) go to stderr.

| Exit code | Meaning |
|---|---|
| `0` | Success |
| `64` | Usage error (missing `--project`, unknown option) |
| `1` | Adapter error (missing required fields, unresolvable `from`, merge conflict, invalid config) |

Keys of the `--config` JSON (they correspond to the adapter entry in the
`adapters:` section of `ductus.config.yaml`):

| Key | Default | Meaning |
|---|---|---|
| `deriveFrom` | `["react-router", "next"]` | Derivation sources (path C) |
| `include` | `["src/**", "app/**", "pages/**", "lib/**"]` | Glob patterns relative to the project that are scanned |

Scanned are files with the extensions `.ts`, `.tsx`, `.mts`, `.cts`, `.js`,
`.jsx`, `.mjs`, `.cjs`. `node_modules`, `dist`, `build`, `out`, `coverage`
and dot directories (`.git`, `.next`, ‚Ä¶) are never scanned ‚Äî regardless of
the `include` globs.

## How @ductus/core finds the adapter

`ductus extract` resolves the command for the adapter entry `typescript`
in this order:

| # | Source | Behavior |
|---|---|---|
| 1 | `command:` in the adapter entry of `ductus.config.yaml` | Always wins ‚Äî the configured command is executed as-is. |
| 2 | `node_modules/.bin` next to the `ductus.config.yaml` | The binary `ductus-adapter-typescript` from `npm install -D @ductus/adapter-typescript`. |
| 3 | `PATH` | The binary from `npm install -g @ductus/adapter-typescript`. |

If no step matches, the call aborts with an error message that names both
installation options. Unlike the Dart adapter, no additional toolchain is
required ‚Äî the adapter runs in the same Node that runs `@ductus/core`.

## Limitations

- **Parse-only means: string literals only.** Paths and targets must be
  statically readable (`'‚Ä¶'`, `"‚Ä¶"` or a template literal without
  interpolation) ‚Äî dynamic paths like `` navigate(`/users/${id}`) `` are
  not recognized, and routes without a literal `path` are skipped with a
  note.
- **`navigate(‚Ä¶)`/`router.push(‚Ä¶)` need the hook context:** `navigate('‚Ä¶')`
  is only recognized in files that use `useNavigate`, `router.push`/
  `router.replace` only in files with `useRouter` ‚Äî free functions with the
  same name are thus not falsely captured.
- **Resolution stops at the file boundary:** route constants
  (`createBrowserRouter(routes)`) and `loader` functions are only resolved
  when declared in the same file ‚Äî imported route arrays or guards
  (`import { requireAuth } from './guards'`) go unrecognized.
- **`@journey:` blocks only in `//` line comments**, not in
  `/* ‚Ä¶ */` block comments.
- **No Vue, Svelte or Angular in this version:** `deriveFrom` knows
  `react-router` and `next`. Path A, however, works framework-agnostic in
  any TS/JS project.

Syntax errors in individual files do not abort the run ‚Äî the TypeScript
compiler API parses fault-tolerantly, the adapter reports the file with a
warning on stderr and keeps analyzing best effort.
